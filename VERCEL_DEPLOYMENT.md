# Vercel Deployment Guide

This project is configured for Vercel serverless deployment with separate frontend and backend functions.

## Architecture

- **Frontend**: React + Vite, deployed as static site to Vercel
- **Backend**: Django serverless function via `@vercel/python`, requires DATABASE_URL
- **Database**: Requires external PostgreSQL database (e.g., Neon, AWS RDS, Supabase, Render-managed)
- **Routing**: `vercel.json` routes `/api/*` and `/payments/*` to Django backend

## Prerequisites

1. **PostgreSQL Database** - Must be external since Vercel serverless can't maintain persistent SQLite
   - Options:
     - **Neon** (recommended): Free tier available, https://neon.tech
     - **Supabase**: Free PostgreSQL tier, https://supabase.com
     - **AWS RDS**: Pay-as-you-go
     - **Render**: Free tier, https://render.com
   - Get your `DATABASE_URL` from the provider (format: `postgresql://user:password@host:port/dbname?sslmode=require`)

2. **Generated Secrets**
   ```bash
   python generate_secrets.py
   ```
   This creates `settings.json` with required secrets. Copy the values for environment variables below.

## Deployment Steps

### 1. Connect Repository to Vercel

1. Go to https://vercel.com/dashboard
2. Click "Add New Project"
3. Import your GitHub repository
4. Vercel auto-detects `vercel.json` configuration

### 2. Configure Environment Variables

In Vercel Project Settings → Environment Variables, add:

#### Required for Backend
- **DATABASE_URL**: `postgresql://user:password@host:port/dbname?sslmode=require`
  - ⚠️ **CRITICAL**: Without this, payments will fail with "schema_not_ready"
  
- **DJANGO_SECRET_KEY**: From `settings.json` (generated by `generate_secrets.py`)
- **DJANGO_DEBUG**: `False`
- **MPESA_ADMIN_TOKEN**: From `settings.json` (generated by `generate_secrets.py`)

#### Optional but Recommended
- **ALLOWED_HOSTS**: `yourdomain.vercel.app,yourfrontend.domain`
- **CORS_ALLOWED_ORIGINS**: `https://yourdomain.vercel.app`
- **MPESA_CALLBACK_URL**: `https://yourdomain.vercel.app/api/payments/callback/`
- **DJANGO_AUTO_MIGRATE**: `True` (enables runtime migrations as fallback)
- **EMAIL_HOST_USER**: Your email for notifications
- **EMAIL_HOST_PASSWORD**: Your email app password

### 3. Deploy

```bash
git push  # Push changes to trigger auto-deploy from Vercel
```

Or manually deploy in Vercel dashboard:
1. Go to Deployments → Import Git Repository
2. Vercel runs `buildCommand` from `vercel.json` (which runs migrations)
3. Frontend and backend functions deploy automatically

### 4. Verify Deployment

1. Check deployment logs in Vercel dashboard for any errors
2. Test payment initiation at `https://yourdomain.vercel.app/payments`
3. Check backend logs if "Payments are temporarily unavailable" error appears

## Troubleshooting

### "Payments are temporarily unavailable" Error

This means the database schema is missing. Check:

1. **DATABASE_URL is set in Vercel environment variables**
   ```
   Vercel Dashboard → Project Settings → Environment Variables
   ```
   
2. **Migrations ran during build**
   - Check Vercel Deployments → Build logs
   - Should see: `Running migrations...` output
   - If migrations failed, they'll run at first request (slower)

3. **Database is accessible from Vercel**
   - Ensure database accepts connections from Vercel's IP ranges
   - Neon and Supabase: Usually allow by default
   - AWS RDS: May need security group rules updated

4. **DJANGO_DEBUG=False in Vercel environment**
   - Prevents fallback to SQLite
   - Ensures real database is used

### Build Fails: "Could not find database module"

Ensure `backend/tastybites/requirements.txt` is present with `psycopg2-binary` listed.

### Function invocation errors

Check Vercel function logs:
1. Go to Deployments → Select most recent
2. Click "View Build Logs"
3. Look for Python traceback or environment variable warnings

## How It Works

### Cold Start Flow (First Request)

1. Vercel invokes `backend/api/index.py` → `app` or `handler`
2. Django initializes with settings from environment variables
3. If `DATABASE_URL` set and migrations pending → runs automatically
4. Request handled → response returned
5. Function execution ends (serverless function is stateless)

### Subsequent Requests

- New cold starts still happen, but cached dependencies speed up initialization
- Database schema persists (external PostgreSQL)
- Migrations only run if new migrations are pending

## Common Issues & Solutions

| Issue | Cause | Solution |
|-------|-------|----------|
| 503 schema_not_ready | Migrations not applied | Check DATABASE_URL is set, wait for retry |
| 500 backend_initialization_failed | Django import error | Check all env vars are set, review build logs |
| 429 Too Many Requests | Rate limiting | Vercel has per-function limits, upgrade plan if needed |
| Payments work locally but fail on Vercel | DATABASE_URL or MPESA token missing | Verify all env vars in Vercel dashboard |

## Performance Notes

- ⚠️ Cold starts can take 5-30 seconds (Django imports, database checks)
- ✅ Warm starts are fast (~100ms)
- ✅ Upgrading Vercel plan adds more concurrent function instances
- ✅ Use HTTP Keep-Alive in frontend code (already configured)
